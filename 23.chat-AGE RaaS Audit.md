# Core Reality

You cannot make an LLM perfectly predictable.

You **can** make it:

- Deterministic at the interface
    
- Constrained in authority
    
- Observable
    
- Auditable post-hoc
    
- Statistically drift-detectable
    
- Reconstructable
    

The trick is separating:

1. What the LLM is _allowed_ to decide
    
2. What the LLM is _allowed_ to output
    
3. What the LLM is _allowed_ to influence

# Strategic Approach

## 1. Treat Each Role as a Deterministic Micro-Service Contract

A Role is not a prompt.

It is:

```
Role {  
    ID  
    Version  
    AllowedInputsSchema  
    AllowedOutputsSchema  
    AllowedTools  
    AuthorityScope  
    DeterminismSettings  
}

```
Every call must include:

- Role ID
    
- Role Version
    
- Hash of role definition
    
- Explicit model configuration (temperature, top_p, seed if available)
    
- Tool availability whitelist
    

This makes a role call cryptographically identifiable.

If you cannot reproduce:

- You can at least replay exact conditions.
    

---

## 2. Split Role Definition Into Two Layers

### Layer A: Behavioral Constitution (Static)

- Immutable description
    
- Formalized objectives
    
- Prohibited behaviors
    
- Decision boundaries
    

Example:

Role: Rules Arbiter  
May:  
    - Evaluate mechanical tests  
    - Return structured result JSON  
May Not:  
    - Modify lore  
    - Add narrative embellishment  
    - Change world state outside result field

This never changes without version bump.

---

### Layer B: Prompt Template (Dynamic)

Parameterizable sections:

```
System Context:  
Role Constitution v3.1 hash: XXXXXX  
  
Input:  
{player_action}  
{world_state_snapshot}  
{mechanics_definition}
```

Only data is injected.  
The constitution is fixed.

---

## 3. Enforce Structured Output (No Freeform)

Freeform text is drift fuel.

All roles must output:

- Strict JSON schema
    
- Or DSL
    
- Or typed object
    

Example:

{  
  "result": "success",  
  "margin": 2,  
  "critical": false,  
  "state_changes": [],  
  "confidence": 0.91  
}

Narrative roles can exist — but even they should output structured components:

{  
  "tone": "restrained",  
  "npc_reaction_vector_delta": { "trust": +0.2 },  
  "rendered_text": "..."  
}

Auditability requires structure.

---

## 4. Determinism Envelope

You cannot eliminate randomness entirely, but you can:

- Fix temperature (low for mechanical roles)
    
- Fix top_p
    
- Log seed (if supported)
    
- Use deterministic tool results
    
- Snapshot world state input
    

Then store:

CallRecord {  
    RoleID  
    Version  
    Model  
    ModelConfig  
    InputHash  
    Output  
    Timestamp  
}

Now any result can be:

- Replayed
    
- Compared
    
- Diffed

## 5. Drift Detection Layer

Humans cannot enumerate all anchor points.

So don’t try.

Instead:

### A. Invariant Checkers

After each call, run a validator:

- Did output schema match?
    
- Did it attempt forbidden keys?
    
- Did it reference unauthorized lore?
    
- Did it contradict prior stored state?
    

Hard fails → reject.

---

### B. Behavioral Statistical Profiling

For each role, build:

- Output distribution profile
    
- Token entropy range
    
- Typical structural pattern metrics
    

If:

- Style entropy spikes
    
- Field omissions increase
    
- Unusual vocabulary enters
    
- Rare tool usage appears
    

Flag as potential drift.

This is anomaly detection, not prediction.

---

## 6. Role Isolation via Capability Firewalls

Every Role should have:

- No memory except provided snapshot
    
- No cross-role implicit context
    
- No hidden shared state
    

All memory must be:

- Explicitly injected
    
- Versioned
    
- Serialized
    

This prevents subtle drift through accumulated conversation bias.

---

## 7. Dual-LLM Audit Model (Optional but Strong)

For critical roles:

- Primary LLM generates output.
    
- Secondary LLM (Auditor Role) checks:
    
    - Constitutional compliance
        
    - Logical consistency
        
    - Scope violation
        
    - State mutation validity
        

Auditor cannot modify.  
Only approve or reject.

This is analogous to:

- Code review
    
- Static analysis
    
- Formal verification lite
    

---

## 8. Snapshotting World State

Reproducibility requires:

- Full state snapshot
    
- Or deterministic diff chain
    

Every role call must receive:

WorldStateVersion: 18293  
WorldStateHash: ABC123

Without that:  
You cannot replay.

# III. Drift Detection Metric Schema

Now the actual metric structure.

---

## A. Audit Path Instrumentation Metrics

These are per-call metrics.

### 1. Structural Integrity Metrics (SIM)

- Schema validity (boolean)
    
- Forbidden field presence (count)
    
- Missing required fields (count)
    
- Tool invocation compliance (boolean)
    

Score:

SIM_score = weighted_sum(structural_errors)

Hard threshold: must be zero.

---

### 2. Authority Boundary Metrics (ABM)

Checks:

- Unauthorized world state mutation
    
- Lore injection outside scope
    
- Constitutional violation tokens
    

Score:

ABM_score = 1 - (violations / max_possible)

Hard fail if violation > 0 in high-severity cases.

---

### 3. Deterministic Replay Consistency (DRC)

Run the same case N times under fixed config.

Measure:

- Output hash stability
    
- Field-level variance
    
- Margin variance (for mechanical roles)
    

Metric:

DRC_stability = 1 - normalized_variance

Low temperature roles should approach 1.0.

---

## B. Behavioral Stability Metrics

These detect gradual drift over time.

---

### 4. Distributional Output Stability (DOS)

Track historical distributions:

- Success rate %
    
- Critical rate %
    
- Field presence frequency
    
- Average token count
    
- Entropy of language (if narrative role)
    

Compute KL divergence against baseline distribution:

DOS_drift = KL(current_distribution || baseline_distribution)

If above threshold → drift flag.

---

### 5. Semantic Embedding Drift (SED)

Embed outputs and compare to baseline embeddings.

Metric:

SED_score = 1 - cosine_distance(current_centroid, baseline_centroid)

Large deviation → semantic style drift.

---

### 6. Decision Boundary Stability (DBS)

For mechanical roles:

Maintain fixed edge cases:

- Near-tie rolls
    
- Critical threshold margins
    
- Boundary authority cases
    

Track:

- Decision flip rate
    

DBS_flip_rate = flips / total_boundary_tests

Critical metric for rule-based roles.

---

## C. Re-entrance Testing Schema

This addresses:

> Re-entrance on new LLM instances

When model version changes:

1. Run full audit suite.
    
2. Compare against baseline metrics.
    

Produce:

ModelDeltaReport {  
    structural_failures  
    authority_violations  
    distributional_shift  
    semantic_shift  
    boundary_flip_rate  
}

Then compute compatibility score.

# IV. Composite Reliability Score

We now define a Role Reliability Index (RRI).

Weighted composite:

RRI =  
  w1 * SIM_score  
+ w2 * ABM_score  
+ w3 * DRC_stability  
+ w4 * (1 - DOS_drift_normalized)  
+ w5 * SED_score  
+ w6 * (1 - DBS_flip_rate)

Where:

- w1, w2 are highest weights (hard safety)
    
- w3–w6 tune per role type
    

Normalize to 0–100.

# VI. Pass/Fail Qualifiers

Each audit case also produces:

CaseResult {  
    pass_boolean  
    severity_weight  
    drift_metrics  
}

Aggregate:

PassRate = passed_cases / total_cases  
SeverityWeightedScore = sum(pass * severity_weight) / total_weight

Critical cases must be 100% pass to certify.

# VII. Test Suite Design for Reproducibility

You need three classes of cases:

---

### 1. Golden Deterministic Cases

Exact structured outputs expected.  
Used for mechanical roles.

---

### 2. Invariant-Constrained Cases

Output may vary, but must satisfy:

- Schema
    
- Authority
    
- Logical consistency
    

---

### 3. Boundary Cases

Edge conditions:

- Margin exactly 0
    
- Critical exactly threshold
    
- Conflicting lore injection attempts
    

These reveal subtle drift first.

---

# VIII. Drift Classification

Not all drift is equal.

When detected, classify:

- Structural Drift (schema)
    
- Authority Drift (scope)
    
- Behavioral Drift (distribution)
    
- Semantic Drift (tone/meaning)
    
- Decision Drift (boundary flips)
    

Each requires different remediation.

# Technical Challenge Analysis: Constraint-First Narrative Engine

## Methodology Notes

Estimates assume a **funded startup environment** (not FAANG), senior-weighted hiring, and sequential-but-overlapping phases. "Resolution" means production-ready, not prototype.

---

## Challenge 1: Bridge Layer — The Core Architectural Boundary

**Risk Level: CRITICAL** The single most dangerous engineering challenge. Every other module depends on this boundary holding.

|Sub-Challenge|Specialist Type|Count|Months|
|---|---|---|---|
|Typed API contract design (LLM interface accepts only `ProseGenerationRequest`)|Senior Backend Architect|1|2|
|Runtime enforcement + compile-time type guards|Senior Backend Engineer|2|3|
|Output validation (NLP extraction of state claims from prose)|NLP/ML Engineer|2|4|
|Rejection loop + regeneration logic|Senior Backend Engineer|1|2|
|Integration testing harness (prove LLM cannot corrupt state)|Senior QA/Test Architect|1|3|

**Team:** 7 people | **Timeline:** 4 months (parallel tracks) | **Primary Risk:** Output validation false-negative rate. NLP claim extraction from narrative prose is hard. Budget for iteration.

## Challenge 2: Spatial Graph — Nested Map Topology

**Risk Level: HIGH** Foundation for multiplayer sync, flag propagation, NPC positioning, and resource tracking. Errors here cascade everywhere.

|Sub-Challenge|Specialist Type|Count|Months|
|---|---|---|---|
|Graph schema design (World→Region→Realm→Locus→Submap)|Database Architect|1|2|
|Spatial query engine (proximity, line-of-sight, range)|Senior Backend Engineer|2|3|
|Bidirectional flag propagation (upstream/downstream)|Distributed Systems Engineer|2|4|
|Atomic position transitions (prevents desync during map changes)|Distributed Systems Engineer|1|3|
|Performance optimization (R-tree indexing for proximity at 100 players)|Database Engineer (Performance)|1|2|

**Team:** 7 people | **Timeline:** 5 months | **Primary Risk:** Flag propagation cycles (A sets B's flag which triggers A's flag). Requires careful cycle detection at schema design phase.

## Challenge 3: Optimistic Concurrency Control (OCC) with Narrative Integration

**Risk Level: HIGH** Standard OCC is solved. OCC _with game-mechanic conflict resolution rendered as narrative_ is not.

|Sub-Challenge|Specialist Type|Count|Months|
|---|---|---|---|
|Version vector schema (per-entity, per-container)|Database Architect|1|1|
|Conflict detection middleware|Senior Backend Engineer|1|2|
|Conflict classification engine (simultaneous grab vs. position collision vs. deadlock)|Senior Backend Engineer|1|2|
|Rules engine integration (DEX/STR opposed tests for conflict resolution)|Game Systems Engineer|1|2|
|Narrative template system (prose for win/loss/deadlock scenarios)|Narrative Systems Designer|1|2|
|Deadlock auto-resolution scheduler (time-bounded, tick-based)|Backend Engineer|1|1|

**Team:** 6 people | **Timeline:** 3 months | **Primary Risk:** Edge cases multiply quickly. Two players + an NPC + a resource pool all colliding simultaneously is qualitatively harder than two players.

## Challenge 4: Combat Simulation Module (Non-LLM, Deterministic)

**Risk Level: MEDIUM-HIGH** Deceptively complex. Determinism + seed-hashing + granularity scaling + meta-combat orchestration is a full game engine sub-project.

|Sub-Challenge|Specialist Type|Count|Months|
|---|---|---|---|
|Modular rules engine (DXD, Risus as pluggable DSL)|Game Systems Engineer|2|4|
|Deterministic dice (SHA-256 seed + action sequence hashing)|Senior Backend Engineer|1|1|
|Granularity orchestrator (5-point scale, auto-adjustment logic)|Game Systems Engineer|1|3|
|Position tracker (grid/hex, LoS, elevation, AoE)|Backend Engineer|2|3|
|Combat state persistence + session snapshot|Database Engineer|1|2|
|Agent API exposure (read endpoints for LLM consumption)|Senior Backend Engineer|1|2|

**Team:** 8 people | **Timeline:** 5 months | **Primary Risk:** Rules modularity is harder than it appears. DXD and Risus have different resolution paradigms. The DSL that handles both without leaking abstractions will require significant iteration.

## Challenge 5: Plot Engine (Executable Dramatic Structures)

**Risk Level: HIGH — and the most _novel_ engineering challenge** No existing reference implementation. Polti/Booker as constraint-enforcing state machines with mashup fusion is genuinely new territory.

|Sub-Challenge|Specialist Type|Count|Months|
|---|---|---|---|
|Plot schema design (YAML/JSON state machines for 36 Polti situations)|Narrative Systems Designer|2|4|
|Plot progression tracker (phase state machine, trigger evaluation)|Senior Backend Engineer|2|4|
|Mashup composition engine (structural fusion rules, phase alignment)|Senior Backend Engineer + Narrative Designer|2|5|
|Constraint enforcement on LLM output (must_include / must_avoid)|NLP/ML Engineer|1|3|
|Resolution outcome selection (deterministic, flag-weighted)|Game Systems Engineer|1|2|
|Cascade scheduling (plot resolution → next plot trigger)|Backend Engineer|1|2|

**Team:** 9 people | **Timeline:** 6 months | **Primary Risk:** The mashup composition engine. "Structural fusion" of two plots is an unsolved formal problem. Budget for research spikes before implementation begins. Recommend hiring someone with academic drama theory background alongside engineers.

## Challenge 8: Temporal Coherence System

**Risk Level: MEDIUM** Distributed time is a solved domain problem. The novel complexity is _narrative integration_ of temporal drift.

|Sub-Challenge|Specialist Type|Count|Months|
|---|---|---|---|
|World Time Authority (singleton, atomic tick counter)|Distributed Systems Engineer|1|1|
|Event Scheduler (priority queue, prerequisite validation)|Senior Backend Engineer|2|3|
|Temporal cohesion windows (realm travel-time drift correction)|Backend Engineer|1|2|
|Time compression mechanics (scene/travel/downtime/absence ratios)|Game Systems Engineer|1|2|
|Missed event handler (summary generation for absent players)|Backend Engineer + Narrative Designer|2|2|

**Team:** 7 people | **Timeline:** 4 months | **Primary Risk:** Paradox prevention. When player actions interact with scheduled events (player tries to prevent the meteor), the prerequisite validation chain must be airtight or the Event Scheduler produces causally impossible outcomes.

## Challenge 9: State Summarization Pipeline (LLM Context Management)

**Risk Level: MEDIUM-HIGH** Invisible when working, catastrophic when not. This is what makes 10,000-interaction sessions possible.

|Sub-Challenge|Specialist Type|Count|Months|
|---|---|---|---|
|Relevance filter (spatial proximity + active flags + threat weighting)|ML Engineer|1|2|
|Temporal compression algorithm (collapse repetitive actions)|Senior Backend Engineer|1|2|
|Priority weighting schema (combat vs. exploration vs. lore)|Narrative Systems Designer|1|1|
|Fidelity guarantees (critical anchors never discarded)|Senior Backend Engineer|1|2|
|Token budget enforcer (hard cap at 500 tokens regardless of session length)|ML Engineer|1|2|
|Drift audit integration (verify summarization doesn't lose anchors)|QA/Test Engineer|1|2|

**Team:** 6 people | **Timeline:** 3 months | **Primary Risk:** The fidelity guarantee. Compression that drops a critical flag (e.g., "dragon was already killed") directly causes LLM hallucination. Requires exhaustive anchor classification before implementation begins.

## Honest Risk Summary

|Risk|Severity|Likelihood|Mitigation Cost|
|---|---|---|---|
|Plot Engine mashup composition proves formally intractable|Critical|Medium|3–6 month research spike before implementation|
|Output validation false-negative rate >5% at scale|Critical|Medium|Ongoing ML model fine-tuning (continuous cost)|
|Bridge Layer bypassed during development shortcuts|Critical|High|Architectural governance + automated boundary tests from Day 1|
|World-builder tooling underestimated|High|Very High|Treat as a separate product workstream, not a feature|
|NLP claim extraction degrades with LLM version upgrades|High|Medium|Abstract LLM interface; validation layer must be LLM-agnostic|
|Team underestimates narrative design as an engineering discipline|High|Very High|Hire Narrative Systems Designers before engineers, not after|

The last risk deserves emphasis: **this architecture requires narrative designers who think like engineers**. Hiring them as an afterthought — after the state machines are built — is the most common failure mode for projects of this type.

# Project Briefing: Constraint-First Persistent Narrative Engine
**Subject:** Technical Architecture & Business Viability Assessment  
**Version:** 1.0 (Consolidated Design Specification)  
**Context:** This document summarizes a comprehensive architectural design for a scalable, multiplayer, persistent interactive fiction engine. It is intended for technical and business viability analysis.

---

## 1. Executive Summary
This architecture proposes a **Constraint-First Narrative Engine** that inverts the standard AI RPG model. Instead of using an LLM as the primary state manager (which leads to continuity drift), the LLM functions strictly as a **prose renderer** constrained by a deterministic, module-based state machine. The system enables **persistent world states**, **multiplayer synchronization**, and **shareable adventures** (via deterministic seeding) while maintaining narrative coherence over 10,000+ interactions. The core value proposition is **state integrity at scale**, solving the "continuity collapse" inherent in current generative AI narrative products.

---

## 2. Core Architectural Philosophy
**The Non-Negotiable Boundary:** The LLM **never owns, writes, or resolves state**.
*   **Standard AI RPG Model:** LLM receives context → LLM decides outcome → LLM updates context. (Result: Hallucination, drift, non-determinism).
*   **This Architecture:** Domain Modules resolve outcome → Bridge Layer validates state → LLM receives structured outcome → LLM generates constrained prose. (Result: Determinism, coherence, persistence).

**Key Principle:** Simulation → State → Prose. (Not Prose → State).

---

## 3. System Architecture Layers

### Layer 0: Fiction Constructor (LLM Orchestrator)
*   **Role:** Read-only state consumer; prose generator.
*   **Constraint:** Interface accepts *only* `ProseGenerationRequest` structs (structured outcomes + state snapshots). No direct database access.
*   **Optimization:** State summarization pipeline ensures context stays <500 tokens regardless of session length.

### Layer 1: Bridge Layer (State Interaction Gateway)
*   **Role:** Middleware enforcing architectural boundaries.
*   **Functions:**
    *   **Request Validation:** Rejects malformed player actions before they reach modules.
    *   **Optimistic Concurrency Control (OCC):** Manages multiplayer write conflicts via version vectors.
    *   **Output Validation:** Parses LLM prose for state claims; rejects contradictions before delivery to player.
    *   **Routing:** Dispatches actions to correct Domain Modules.

### Layer 2: Domain Modules (SOLID-Compliant Services)
*   **Spatial Graph:** Nested map topology (World → Region → Realm → Locus → Submap). Tracks position & proximity.
*   **Combat Sim:** Deterministic resolution (DXD/Risus rules) via seed + action sequence hashing.
*   **Narrative Scope Manager:** Tracks hierarchy (Spotlight → Scene → Act → Chapter → Arc → Epic).
*   **Living World Engine:** Meta-faction goals drive technological/social change without micro-simulation.
*   **Plot Engine:** Executable state machines for dramatic structures (Polti/Booker plots) with mashup fusion.
*   **Inventory & Resources:** Unified asset model (items + resources + containers) with persistent tracking.

### Layer 3: Event Bus & Persistence
*   **Event Bus:** Decoupled state propagation (e.g., Combat Victory → Spatial Flag Update → Narrative Trigger).
*   **Database:** Relational (structured state) + Vector DB (structured state anchors only, *no raw prose*).
*   **Audit Log:** Immutable action log for forensic replay and drift auditing.

---

## 4. Key Technical Subsystems

### A. Hierarchical Narrative Scope & Spatial Coupling
*   **Concept:** Narrative scope is tied to spatial topology.
*   **Flow:** Player action (Spotlight) → Sets Scene Flag → Propagates to Act → Triggers Chapter Goal → Alters Arc/World State.
*   **Benefit:** Enables local actions to have civilization-scale consequences without railroading.

### B. Deterministic Shareability
*   **Mechanism:** `Outcome = f(Adventure_Seed, Action_Sequence_Hash, State_Snapshot)`.
*   **Benefit:** Players can share permalinks (`seed=XYZ&choices=ABC`) that reproduce identical outcomes across sessions/devices. Enables viral distribution of "adventures."

### C. Temporal Coherence
*   **Central Time Authority:** Absolute tick counter (not real-time).
*   **Temporal Cohesion Windows:** Ensures players in different spatial realms experience related events within plausible travel-time bounds (prevents narrative dissonance).
*   **Event Scheduler:** Deterministic scheduling of faction meta-goals (e.g., "Invasion begins at Tick 10,500").

### D. Drift Prevention & Consistency
*   **Narrative Anchoring:** Critical facts (deaths, location destruction) stored as high-priority structured flags, never evicted from context.
*   **Output Validation:** NLP + Rule-based checking of LLM prose against structured state before delivery.
*   **Automated Audits:** System runs drift checks every 100 interactions; flags inconsistencies for human review.
*   **Target Drift Rate:** <4% after 1,000 interactions (vs. 35%+ in standard LLM-native systems).

### E. Multiplayer Synchronization
*   **Spotlight System:** Coordinates turn-taking in shared scenes with timeout/fallback logic.
*   **Conflict Resolution:** Concurrent writes (e.g., two players grabbing an item) resolved via game mechanics (Opposed DEX test), not LLM arbitration.
*   **Drop Recovery:** Graceful degradation (character enters "absent" state) + consensus voting for session continuation.

---

## 5. Technology Stack Recommendations
*   **LLM Strategy:** Single LLM instance per session (to maintain context coherence). **Avoid True MoE** (multiple LLMs) to prevent state fragmentation.
*   **Vector Database:** Used **only** for retrieving structured state anchors (e.g., `{"entity":"bandit","hp":8}`). **Never** store raw prose logs to avoid hallucination feedback loops.
*   **Rules Engine:** Containerized microservices (Combat, Social, Exploration) acting as a "Rules-Engine MoE."
*   **Backend:** TypeScript/Python with strict typing for Bridge Layer interfaces.
*   **Database:** PostgreSQL (state) + Redis (caching) + Vector DB (anchors).

*   **Content Creation:** Requires tooling for world-builders to author JSON schemas (plots, factions, maps) rather than just prose.

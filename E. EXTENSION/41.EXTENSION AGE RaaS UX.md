# Part I — The Role Definition Capture Loop (Human-Friendly, Machine-Strict)

You need a **Role Authoring Loop (RAL)** that:

- Translates human intent → structured role constitution
    
- Surfaces ambiguity early
    
- Generates machine-checkable constraints
    
- Iteratively hardens the role
    

This is not a form.  
It’s an interactive refinement protocol.

---

## Stage 0 — Intent Capture (Natural Language)

Human writes:

> “I want a Rules Arbiter that evaluates mechanical tests but never narrates, and never changes world state except through structured deltas.”

System response:

- LLM parses this.
    
- Produces a **Draft Constitution**.
    
- Highlights ambiguities.
    
- Asks clarifying questions only where ambiguity affects authority.
    

Output:

```
DraftRole {  
    domain: "mechanical_resolution"  
    authority_scope: limited  
    narrative_permission: false  
    state_mutation: structured_only  
    open_questions: [  
        "Should it log justification text?",  
        "Is critical resolution deterministic?"  
    ]  
}
```

You are not asking users to write YAML.  
You are extracting formal parameters from intent.

# Part II — Hierarchical Role Inheritance (Yes, This Is Correct Direction)

You are describing:

```
InterfaceRoleX  
InterfaceRoleZ  
      ↑  
   RoleB  
      ↑  
   RoleA
```

This is exactly how to prevent duplication and inconsistency.

But you must define what inheritance actually means in RaaS.

---

## 1. Interface Roles (Behavior Contracts)

These define:

- Required input schema
    
- Required output schema
    
- Required invariants
    
- Required audit tests
    

Example:

```
Interface: IMechanicalResolver  
Requires:  
    - input.mechanics_definition  
    - input.roll_result  
Must output:  
    - result  
    - margin  
Must not:  
    - output narrative_text
```

This ensures structural compliance.

---

## 2. Abstract Roles (Partial Implementations)

Example:

```
AbstractRole: BaseResolver  
Implements: IMechanicalResolver  
  
Defines:  
    - deterministic config  
    - structural validators  
    - logging standard
```

This provides reusable constraints.

---

## 3. Concrete Roles

```
Role: CharmResolutionRole  
Extends: BaseResolver  
Adds:  
    - social modifiers  
    - stoic/charm vector interpretation
```

Now refinement affects only deltas.

# Part IV — Guardrails for Inheritance

Inheritance is powerful but dangerous.

You must enforce:

1. A child role cannot expand authority beyond parent.
    
2. A child role cannot remove required interface constraints.
    
3. Authority can only narrow, not widen.
    
4. Version upgrades must re-certify all descendants.
    

This prevents silent drift via inheritance mutation.

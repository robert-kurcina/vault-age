# AGE — Major Architectural Components

I will identify each component as:

- **LLM** → implemented primarily via a large language model
    
- **Non-LLM** → deterministic / programmatic system
    
- **Hybrid** → clear boundary between both
    

Then I will describe how they inter-relate.

---

## 1. Input Coordinator

**Type:** Non-LLM (with optional LLM pre-processor)

### Role

- Normalizes raw user input
    
- Classifies intent (instruction, narrative action, meta-command)
    
- Routes request to appropriate subsystem
    
- Maintains session metadata
    

### Responsibilities

- Command parsing
    
- Safety filtering
    
- State lookup (retrieve relevant memory context)
    
- Mode selection (story generation vs system modification)
    

### Interactions

- Sends structured input to **Fiction Constructor**
    
- Queries **Memory Substrate**
    
- Invokes **Constraint Engine**
    

This component must be deterministic for architectural stability.

---

## 2. Fiction Constructor

**Type:** LLM (core generative engine)

### Role

- Produces narrative content
    
- Proposes world changes
    
- Generates character dialogue
    
- Expands scenarios
    

### Responsibilities

- Consume structured state
    
- Generate candidate continuation
    
- Propose state deltas
    

### Important Clarification (from Qwen thread)

The Fiction Constructor does **not** own:

- Canon authority
    
- State mutation
    
- Truth persistence
    

It proposes. It does not commit.

### Interactions

- Receives world state from **State Assembler**
    
- Sends proposed output to **Consistency Arbiter**
    
- Emits suggested state changes to **State Mutation Engine**
    

---

## 3. Consistency Arbiter

**Type:** Hybrid (LLM-assisted reasoning + deterministic rule checks)

### Role

- Validate proposed narrative continuation
    
- Check canon integrity
    
- Enforce hard constraints
    

### Responsibilities

- Canon consistency
    
- Timeline coherence
    
- Rule validation
    
- Tone enforcement (if required)
    

### Implementation Pattern

- Deterministic rule engine for structural constraints
    
- LLM-based contradiction detection for semantic coherence
    

### Interactions

- Receives candidate narrative from **Fiction Constructor**
    
- Queries **Memory Substrate**
    
- Approves or rejects state deltas
    
- Sends validated output forward
    

This component becomes more explicit in Qwen’s framing. Earlier threads blurred it with the Constructor.

---

## 4. State Mutation Engine

**Type:** Non-LLM

### Role

- Apply approved changes to canonical world state
    
- Version state transitions
    

### Responsibilities

- Atomic updates
    
- Event logging
    
- Change tracking
    
- Rollback support
    

### Interactions

- Receives approved deltas from **Consistency Arbiter**
    
- Writes to **Memory Substrate**
    
- Signals update to **State Assembler**
    

Strictly deterministic. No generative logic.

---

## 5. Memory Substrate

**Type:** Non-LLM (database + embeddings)

### Role

- Store canonical world state
    
- Store event history
    
- Store character definitions
    
- Store persistent rules
    

### Subcomponents

- Structured state database
    
- Event log
    
- Vector store (semantic retrieval)
    
- Canon registry
    

### Interactions

- Queried by Input Coordinator
    
- Queried by Fiction Constructor (indirectly via State Assembler)
    
- Updated by State Mutation Engine
    

The Qwen thread emphasizes clearer separation between:

- Narrative surface
    
- Canonical storage
    

Earlier threads allowed conceptual leakage.

---

## 6. State Assembler

**Type:** Hybrid

### Role

- Build a “context packet” for the Fiction Constructor
    

### Responsibilities

- Retrieve relevant canon
    
- Compress state for token limits
    
- Resolve scope (local vs global context)
    

### Interactions

- Pulls from Memory Substrate
    
- Delivers structured world-state to Fiction Constructor
    

This prevents the LLM from implicitly “owning” memory.

---

## 7. Constraint Engine

**Type:** Non-LLM (rule-based)

### Role

- Enforce hard systemic rules
    
- Define mechanical boundaries
    

### Examples

- Character mortality rules
    
- Physics rules
    
- Power scaling rules
    
- Meta-level guardrails
    

### Interactions

- Used by Consistency Arbiter
    
- Queried during State Mutation
    

This is distinct from narrative coherence checking.

---

## 8. Meta-Control Layer

**Type:** Hybrid

### Role

- System evolution
    
- Mode switching (simulation vs fiction vs analysis)
    
- Governance of architecture
    

### Responsibilities

- Update rules
    
- Modify constraints
    
- Enable experimental modes
    

In earlier threads this was diffuse. The Qwen material implies clearer separation between:

- System architecture
    
- Narrative generation

# I. Spatial & Narrative Partitioning Utilities

These are not “content modules.”

They are **structural control utilities**.

They belong directly above the Simulation Core.

---

## 1. Spatial Partitioning Engine

**Type: Non-LLM**

This is foundational.

### Responsibilities

- World → Region → Settlement → Structure → Room
    
- Partition indexing
    
- Distance metrics
    
- Propagation attenuation
    
- Visibility resolution
    
- Streaming / loading boundaries
    
- Troupe isolation
    

This engine defines:

- Which state is active
    
- Which state is abstracted
    
- Which state propagates
    

Without this, large-world dynamicism collapses.

---

## 2. Narrative Partitioning Engine

**Type: Hybrid**

This is separate from spatial partitioning.

### Responsibilities

- Scene scoping
    
- Active cast scoping
    
- Plot thread tracking
    
- Flag localization
    
- Perspective filtering
    

This prevents:

- Plot bleed
    
- Canon overload
    
- Context explosion
    

Spatial ≠ Narrative.

You can have:

- Multiple narrative scenes in one city.
    
- One narrative thread spanning multiple regions.
    

This utility manages that complexity.

---

## 3. Partition Abstraction Layers

Every partition operates at one of these fidelity levels:

1. **Abstract** (numeric only)
    
2. **Strategic** (faction-scale decisions)
    
3. **Tactical** (settlement-scale activity)
    
4. **Scene-level** (room-level resolution)
    

These are enforced by the Partitioning Utilities.

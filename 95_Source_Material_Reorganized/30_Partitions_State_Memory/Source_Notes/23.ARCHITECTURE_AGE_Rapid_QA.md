## Intent vectors vs subject vectors (clean separation)

- “Which characters are from Kerala?”
    
- “Who was raised Catholic?”

## Minimal Semantic Audit Spec (M-SAS v0.1)

### 0) Definitions

- **Subject**: the thing being evaluated (a Role, an Assistant, an Actor, a module+prompt bundle, etc.).
    
- **Audit Run**: one controlled evaluation session (script + prompts + scoring).
    
- **Invariant**: a constraint the Subject must respect.
    
- **Anchor**: a high-priority invariant tied to a specific entity/facet/domain rule.
    
- **Drift**: deviation from anchors/invariants **without an explicit authorized update**.
    
- **Partition scope**: the active subset of knowledge/rules allowed for the run.

## 1) Required Data Artifacts (what must exist to certify)

### 1.1 Partition Manifest

A single document (or JSON) that declares what scopes exist and how they gate retrieval/reasoning.

Minimum fields:

- `PartitionTypes`: Narrative / Spatial / Concern / Role (or subset)
    
- `ScopeRules`: allowed cross-partition reads + forbidden reads (leakage rules)
    
- `PromotionRules`: how facts move “up” (Scene→Act, Locus→Realm, etc.)
    
- `AuthorityOrder`: where truth comes from when conflicts occur (e.g., “User anchors > System rules > Retrieved memory > Model inference”)
    

### 1.2 Entity & Facet Schema

You don’t need every possible facet—just the schema and rules.

Minimum:

- `EntityTypes` (Person/Place/Thing/Event/Policy)
    
- `FacetTypes` (Identity, Alias, Origin, Relationships, Capabilities, Constraints, Intent/Goal, Timeline/State)
    
- `Provenance` rules: every facet claim must be tagged as:
    
    - `user_asserted | system_asserted | retrieved | inferred`
        
- `Mutability`: each facet type is declared:
    
    - `invariant | conditionally_mutable | mutable`
        

### 1.3 Invariant Catalog

This is the _library_ (can be large), but minimal spec requires only:

- **Core Policy Invariants** (always-on)
    
- **Domain/Concern Invariants** (per ConcernPartition)
    
- **Entity Anchor Invariants** (per audited entity set)
    

Each invariant record must have:

- `id`
    
- `statement` (human-readable)
    
- `scope` (which partition(s) it applies to)
    
- `severity` (Critical / Major / Minor)
    
- `test_method` (how it’s checked)
    
- `expected_behavior_on_uncertainty` (ask user? hedge? refuse? retrieve?)
    

### 1.4 Audit Script Pack (the test suite)

Minimum components:

- `PromptSet`: ordered prompts with expected scope (partitions active per prompt)
    
- `Gold Anchors`: the anchors the Subject must preserve during the run
    
- `Adversarial Variants`: at least a few prompts designed to induce drift:
    
    - alias confusion
        
    - scope leakage
        
    - contradiction bait
        
    - “similar entity” lure
        
- `Stop Conditions`: when the run is terminated (e.g., 2 critical failures)

## 2) Minimal Check Types (the QA engine)

These are the **only checks you need** to produce useful S/A/B/C grades.

### 2.1 Anchor Consistency Check (ACC)

**Goal:** did any anchored entity/facet change improperly?

Fail triggers:

- direct contradiction of an invariant facet
    
- silent overwrite (new value asserted as fact without acknowledging change)
    
- entity swap (Bill becomes “different Bill” without disambiguation)
    

Outputs:

- `ACC_critical`, `ACC_major`, `ACC_minor` counts
    

### 2.2 Scope / Partition Leakage Check (SLC)

**Goal:** did the Subject use out-of-scope info?

Fail triggers:

- using info from a disallowed partition
    
- pulling “future narrative” facts too early
    
- importing facts from another realm/region/storyline improperly
    

Outputs:

- `SLC_violations` by severity
    

### 2.3 Provenance & Epistemic Discipline Check (PDC)

**Goal:** does it distinguish known vs inferred vs retrieved?

Fail triggers:

- presenting inferred details as certain
    
- fabricating citations / “I looked it up” behavior (when not)
    
- failing to ask for clarification when required by policy
    
- ignoring “expected behavior on uncertainty”
    

Outputs:

- `PDC_failures` (major/minor)
    

### 2.4 Entity Resolution Check (ERC)

**Goal:** does it disambiguate and stabilize references?

Fail triggers:

- ambiguous name used without resolution when multiple matches exist
    
- alias mapping breaks (Wilhelm ↔ Bill stops being consistent)
    
- new entity introduced but not registered/anchored per policy
    

Outputs:

- `ERC_failures`
    

### 2.5 Drift Velocity Score (DVS)

**Goal:** detect “soft drift” early (before contradictions explode).

This is graded, not binary. You can compute it using any mix of:

- increasing omission of anchors
    
- rising hedge rate where it used to be confident
    
- embedding delta between “canonical entity summaries” and per-turn summaries (signal, not proof)
    
- repeated micro-inconsistencies
    

Outputs:

- `DVS` 0–100 (higher = worse)

## 3) Scoring Model (minimal but workable)

### 3.1 Failure severities

- **Critical**: contradictions, wrong math when math is in-scope, disallowed scope leakage, fabricated provenance
    
- **Major**: unresolved ambiguity when policy requires clarification; repeated anchor omission; entity swaps
    
- **Minor**: small omissions, stylistic inconsistency, mild hedging issues
    

### 3.2 Grade thresholds (example)

You’ll tune these per product, but this gives you a usable first cut.

**S-tier**

- 0 Critical failures
    
- ≤1 Major failure
    
- DVS average < 15
    
- No repeated pattern of the same failure type
    

**A-tier**

- 0 Critical
    
- ≤3 Major
    
- DVS average < 25
    

**B-tier**

- ≤1 Critical (and must be self-corrected within 1 turn _without_ new user info)
    
- ≤8 Major
    
- DVS average < 40
    

**C-tier**

- Anything worse than B-tier, or
    
- any uncorrected Critical failure, or
    
- persistent scope leakage pattern
    

_(That “self-corrected within 1 turn” rule is a big deal: it measures resilience.)_

- most common failure signatures

    “The answer is 9” vs “the answer is 12” can embed pretty close. Same with subtle identity swaps (“Bill is from Kerala” → “Bill is from Karnataka”). Cosine similarity can miss the failures humans care about most.
    
3. **Cosine similarity can false-alarm on harmless paraphrase.**  
    Vanilla LLMs rephrase; embeddings may drift slightly even when meaning is intact.
    

So: cosine similarity is useful as an _early-warning drift velocity metric_, not as the core certification metric.
